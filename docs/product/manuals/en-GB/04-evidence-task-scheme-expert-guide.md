# Evidence and Task Scheme Expert Design Guide

## 1. Design model

The system is a framework for experts to design assessment networks. The installed nodes are starter content, not protected scientific truth. Experts may create many globally shared nodes, then let each task scheme activate the combination appropriate to that task.

Each complete node has one identity, one canonical English name, one definition and one fixed parent set. A node can be shared by several schemes. If a new task requires different parents, parameters or meaning, copy the node, rename the copy and edit it; do not make one node behave differently according to the currently selected scheme.

Examples of parallel complete nodes are `Trajectory Precision`, `Hover Trajectory Precision` and `Straight Flight Trajectory Precision`. They may be similar, but the system treats them as distinct objects. This preserves every task design without manual switching or overwriting.

## 2. Read the five-layer canvas

[[SCREENSHOT:ui-five-layer-model-studio]]

The main canvas is arranged from left to right:

1. **Raw Input** — the five source families `X(t)`, `U(t)`, `I(t)`, `G(t)` and `P(t)`;
2. **Extracted Data** — reusable intermediate signals or event/segment products;
3. **Evidence** — task-observable quantities and their observation mappings;
4. **Sub-skill** — latent BN capability nodes;
5. **Competency** — aggregate BN capability nodes.

Raw Input nodes are larger and green. Other node types use their own visual categories. Operator implementations are not graph nodes: an Evidence or Extracted Data editor shows the operator recipe it uses.

Use the type filter to isolate any of the five layers. Pan and zoom the viewport; hold the primary button and move a node at least 4 px to change its stored canvas position, while an ordinary click still selects it. Layout is presentation metadata; moving a node does not change its fixed parents.

## 3. Create, copy and delete nodes

Use **New node** and choose its layer. Enter a concise canonical English name, short name and description. Technical IDs are generated by the backend and remain in diagnostics/provenance rather than ordinary labels.

Copy and paste is the preferred way to specialise starter content:

1. select the source node and copy it;
2. switch to the target task scheme;
3. paste to create a new complete node;
4. rename and edit the new definition;
5. enable the copy and disable the old node in that scheme.

By default, a copied node continues to reference the original fixed parents. Edit its parent set only when it is genuinely a different complete node. Global deletion is stronger than task deactivation. Select a node and use **Delete node from system model** on the toolbar or context menu. After confirmation, one staged backend transaction removes the node and dependent downstream nodes from every affected scheme's active closure/output, then archives the global node. The status reports the number of affected schemes. Ctrl+Z can undo it; only Save All changes the canonical system and Discard All restores the earlier state. Historical RunSnapshots remain immutable.

## 4. Edit an Evidence node

[[SCREENSHOT:ui-evidence-node-editor]]

Double-click or open a node to create an independent movable, resizable floating editor. Several editors may be open side by side. Editable sections include:

- canonical English name, short name, description and expert help text;
- fixed Raw Input or Extracted Data parents;
- EvidenceRecipe operator graph and typed parameters;
- time/event/window selectors and missing-input policy;
- continuous output definition and units when declared by the method;
- observation states and continuous-to-state mapping;
- trace/provenance presentation metadata.

The UI label language can switch between Chinese and English, but persisted model content remains one canonical English value. Do not enter a second translated node name into the model.

An Evidence node may depend only on Raw Input and/or Extracted Data products. It must not depend on Sub-skill or Competency nodes. The computation path is data to Evidence; Bayesian inference later conditions latent capabilities on that observation.

## 5. Understand recipes and operators

An EvidenceRecipe is a declarative calculation assembled from registered operators. Operators are reusable mechanisms such as selecting an event interval, resolving a signal, measuring AOI dwell, computing error or aggregating a window. Parameters belong to the node recipe and can normally be edited in the front end without changing Python.

For example, an Evidence for whether the pilot looked at a required instrument after a disturbance can:

1. select a disturbance interval from annotations;
2. resolve gaze/AOI samples for that interval;
3. test membership in the target AOI;
4. calculate first-fixation latency or dwell duration;
5. map the continuous value to D/A/U states.

If the existing operator catalog cannot express a genuinely new mechanism, add Python source as described in [[DOC:PAS-PYTHON-EXT-001]], restart the application, then reference the new operator from the recipe.

## 6. Design a task scheme

A task scheme stores a name, description, activation set, active edges and presentation layout references. It does not own a private copy of each node. Schemes are parallel, continuously editable selections in the left sidebar; there is no separate publish lifecycle.

Create a scheme by copying an existing scheme or starting with a new selection. Scheme copying copies the selection and layout, not the global node objects. Later edits to a shared node affect every scheme that activates that same node; historical runs are unaffected because they contain frozen snapshots.

When activating a child, all fixed ancestors are activated automatically without a prompt. When deactivating a parent that has active descendants, the UI must show the affected descendants and ask whether to continue or cancel. Continuing deactivates the dependent downstream closure in that scheme.

Edges are derived from complete-node fixed parents and scheme activation. A scheme cannot override a node's parents. To model a different relation, copy the child node and edit the copy's fixed parents.

## 7. Stage, undo and save changes

Model edits are staged in one durable system-level edit session. They are not written into the current user project and are not committed one field at a time.

- `Ctrl+Z` undoes the latest staged command; redo restores it when available.
- Main-toolbar **Save All** or `Ctrl+S` first flushes floating editors and pending layout, then atomically commits the staged model while the software stays open; future runs in every project use the new model.
- **Discard all** restores the last saved model.
- Closing with changes asks whether to save and close, discard and close, or cancel.

If a save detects a stale optimistic revision, reload or rebase rather than overwriting a concurrent saved state. Run creation requires a clean saved model so its snapshot is unambiguous.

Normal nodes and the five green Raw Input Family roots can all be dragged with the held primary button and must remain at the released position. Green roots are stored only as display layout for the current task scheme; they never become ModelNodes or enter Evidence/BN/CPT/RunSnapshot semantics.

## 8. Scientific review responsibility

The software validates contracts, acyclicity, parameter types and executability. It does not decide whether an Evidence definition measures the intended human skill, whether its thresholds are fair or whether a task scheme is suitable for training decisions.

Record the intended task, operational event, population, units, rationale and review owner in the node description/help text and project documentation. Use controlled datasets and expert review before changing `formal_run_authorized` outside this engineering candidate.

## 9. Expert checklist

- [ ] One complete meaning and one fixed parent set per node;
- [ ] canonical model text is English;
- [ ] task-specific variation uses a copied distinct node;
- [ ] Evidence parents are only Raw Input or Extracted Data;
- [ ] operator parameters and observation mapping are explicit;
- [ ] scheme ancestor closure is complete;
- [ ] downstream impact reviewed before deactivation;
- [ ] staged changes inspected and saved atomically;
- [ ] scientific rationale recorded separately from software validation.
